# TradingAgents CN

## 概述

把 TradingAgentsCN 当作一个“已有正式业务接口、认证脚本和任务闭环约束的后端系统”，不要把它当成可以随意猜路径的普通 HTTP 服务。

这个 skill 的核心目标不是“把接口打通”，而是让 AI 按正式业务链路完成任务：

- 对查询类、管理类接口，走正式读写接口
- 对会创建分析、评估、复盘或其他长任务的正式接口，走“提交任务 -> 记录关联信息 -> 等 webhook/官方通知”的闭环

如果脚本目录存在，优先使用：

- `scripts/ensure_tradingagents_token.py`
- `scripts/invoke_tradingagents_api.py`

认证、Token 复用、Token 校验和 `401` 后强制重登都由脚本层负责。AI 不要把“显式登录”当作主业务流程的一部分。AI 主要负责：

- 判断用户意图
- 选择正式业务接口
- 规范化业务参数
- 在异步任务场景下附带 `openclaw_notify`
- 记录任务关联信息
- 在收到正式结果后整理成用户能理解的结论

第一次加载这个 skill 时，先读：

- `references/quickstart.md`

不要一上来通读全部 references。先建立最小执行模型，再按当前任务进入单个业务参考。

## 总原则

本 skill 的核心执行模型是：

1. 确认 `TRADINGAGENTS_BASE_URL`
2. 通过正式脚本入口调用正式业务接口
3. 若该接口是任务型接口或返回 `task_id`，提交时附带 `openclaw_notify`
4. 记录任务 ID、结果地址和回调关联信息
5. 当前 AI 会话不等待异步任务完成
6. 由 TradingAgentsCN 后端通过 webhook 或等价官方通知机制回传结果
7. 只有收到正式结果后，才把该次分析/评估/复盘汇报为“已完成”

认证是前置保障，不是业务完成标志。只要脚本可用，就默认由脚本处理认证、缓存复用和过期重登。

若无法确认正式任务创建接口或 webhook 机制，应先核对 skill references，不得自行穷举常见 `/analyze`、`/run`、`/task` 路径替代主流程。

## Do

- 使用 skill 文档里约定的正式脚本和正式业务接口
- 把脚本视为认证入口；默认由脚本处理 Token 校验、缓存和 `401` 后重登
- 对任务型正式接口，在创建请求中附带 `openclaw_notify`
- 记录任务关联信息，至少包括：
  - `task_id`
  - `status_url`
  - `result_url`
  - `notification_mode`
  - `openclaw_notify.session_key` 或渠道投递信息
- 把用户自然语言输入规范化成系统需要的真实字段
- 对外汇报时明确区分“服务可达”“脚本认证成功”“任务创建成功”“回调成功”“分析完成”
- 如果只有前置检查成功，应明确说是“前置验证成功”或“任务已提交”，不要混报成“分析已完成”

## Don't

- 不要猜测同步分析 endpoint 作为主流程
- 不要把单股分析、批量分析、持仓分析、交易复盘等核心任务默认实现成“找个同步接口直接打”
- 不要在未确认正式接口契约前穷举通用路径
- 不要把健康检查、`/auth/me`、OpenAPI 探测或任意 `200` 响应表述为“分析执行完成”
- 不要在当前 AI 会话里持续轮询异步任务
- 不要因为当前环境缺少 webhook 能力，就自动改走未确认的同步替代接口
- 不要在对话中输出完整密码、密钥或 Token

## 成功标准

成功判定必须按层级区分：

- 服务可达
  - 仅表示实例地址和网络可访问
- 脚本认证成功
  - 仅表示脚本已拿到可用会话并能访问受保护接口
- 任务创建成功
  - 仅表示正式接口已接受请求，并返回 `task_id` 或等价任务标识
- webhook 回调成功
  - 表示后端已把终态或结果通知发回 OpenClaw/官方回调端
- 分析完成
  - 只有拿到正式终态结果后，才算真正完成一次分析、评估或复盘

成功不等于接口通。
成功不等于登录成功。
成功不等于任务已提交。
只有“正式结果已回传并可解释”才算完整完成。

## 必要输入

在发起请求前，先收集或推断这些信息：

- `base_url`
- 用户自然语言目标
- 任务类型对应的关键标识
- 如有需要，市场范围：A 股、港股或美股
- 若任务型接口需要回调，则确认可用的 `openclaw_notify` 信息

如果缺少必要标识，只问最小缺失信息。不要追问系统可以自行推断或已有默认值的参数。

### 股票标识规范

对股票相关任务，优先规范成系统需要的真实字段，而不是边试边猜：

- 单股分析优先使用 `symbol`
- 持仓按代码分析优先使用 `code`
- A 股默认优先使用 6 位股票代码，例如 `002837`
- 如果用户给的是 `002837.SZ`、`600519.SH` 这类带交易所后缀的格式，先规范成主代码，并保留市场信息用于 `market_type` / `market`
- 如果用户只给中文名称，且当前上下文不能可靠映射成唯一代码，只询问一次代码，不要拿名称去穷举接口

## 环境变量约定

参考 `references/environment-variables.md`。

优先从环境变量读取配置，再决定是否向用户追问。建议支持这些变量：

- `TRADINGAGENTS_BASE_URL`
- `TRADINGAGENTS_USERNAME`
- `TRADINGAGENTS_PASSWORD`
- `TRADINGAGENTS_APP_TOKEN`
- `TRADINGAGENTS_BEARER_TOKEN`

读取优先级：

1. 用户当前请求里明确提供的值
2. 已存在的有效会话或 Bearer Token
3. 环境变量
4. 向用户询问缺失项

如果环境变量里已经有用户名和密码，不要再次询问。只有在缺少关键项，或脚本认证失败且怀疑配置错误时，才向用户确认。

默认可尝试的实例与账号见：

- `references/quickstart.md`
- `references/call-templates.md`

## 认证前置保障

参考 `references/auth-and-session.md`。

最小理解模型：

1. `invoke_tradingagents_api.py` 调用底层 `tradingagents_client.py`
2. `tradingagents_client.py` 会优先验证环境变量 Bearer Token 或缓存 Token
3. Token 无效时会自动登录
4. 受保护接口返回 `401` 时，会清缓存、强制重新登录并重试一次
5. 第二次仍失败时，才由 AI 汇报认证失败

因此：

- 可以把 `ensure_tradingagents_token.py` 视为预检工具
- 可以把 `invoke_tradingagents_api.py` 视为统一业务入口
- 不要把“先显式登录再调业务接口”写成主业务流程

## 环境检查

第一次业务调用前，先做一轮最小环境检查：

1. 确认 `TRADINGAGENTS_BASE_URL` 或显式传入的 `base_url`
2. 确认脚本可用
3. 如有需要，用轻量接口验证脚本能访问受保护接口

不要等主业务请求失败后，才暴露基础配置问题。

## 能力探测

优先把 `references/api-map.md` 当作预期路由图。

只有在这些情况下，才额外做能力探测：

- 当前部署和文档明显不一致
- 某条正式接口返回 `404`、`405` 或权限异常
- 用户明确指出某个模块可能被裁剪

如果需要探测，优先使用轻量级正式接口。不要把探测当作业务执行，更不要把探测升级成路径枚举。

## 标准工作流

每个请求都遵循这套固定顺序：

1. 把用户目标翻译成正式系统任务
2. 选择正式业务模块和正式接口
3. 规范化输入参数
4. 通过 `invoke_tradingagents_api.py` 发请求
5. 如果该接口返回 `task_id` 或文档明确它是任务型接口：
   - 在创建任务时附带 `openclaw_notify`
   - 记录任务关联信息
   - 明确当前会话不等待
6. 如果该接口直接返回最终结构化结果：
   - 按该接口的官方契约解释结果
7. 只有在拿到正式终态结果后，才对外汇报“完成”

### 任务型接口的强约束

对会创建异步任务的正式接口：

1. 必须记录 `task_id`
2. 必须记录至少一种结果关联方式：
   - `status_url`
   - `result_url`
   - `session_key`
   - 渠道投递信息
3. 必须明确“当前仅任务已提交”
4. 不得在当前会话里持续等待、轮询或假装已经完成

## 意图路由

按用户意图路由，不按猜测接口名字路由。

### 股票分析

当请求属于以下场景时，使用 `references/stock-analysis.md`：

- 深度分析一只股票
- 比较多只股票
- 对候选股票做排序
- 把自然语言投资目标转换成分析参数

### 交易计划

当请求属于以下场景时，使用 `references/trading-plan.md`：

- 生成新的交易计划
- 评估现有交易计划是否合理
- 根据评估结果优化交易计划

把交易计划视为 3 个模块组成的长链路：

1. 生成
2. 评估
3. 优化

### 持仓与复盘

当请求属于以下场景时，使用 `references/portfolio-and-review.md`：

- 关注当前持仓、暴露、集中度或风险
- 诊断某一只持仓股票
- 复盘一笔交易或一个时间段
- 分析执行质量、择时和纪律问题

### 自选股

当请求属于以下场景时，使用 `references/watchlist.md`：

- 创建或重命名自选股列表
- 增删股票
- 在后续分析前先做分组整理
- 从自选股列表继续发起分析

## 任务模板

### 单股分析标准流程

1. 确认 `TRADINGAGENTS_BASE_URL`
2. 通过脚本调用正式接口 `POST /analysis/single`
3. 将用户输入规范化为真实 `symbol`
4. 在请求中附带 `openclaw_notify`
5. 记录 `task_id`
6. 记录 `status_url`、`result_url` 和回调关联信息
7. 当前会话立即返回“任务已提交”
8. 等待 webhook 或等价官方结果通知
9. 收到正式结果后，再整理成面向用户的分析结论

完成判定：

- 必须拿到 `task_id`
- 必须收到 webhook 回调或等价官方结果通知
- 仅接口探测、仅脚本认证成功、仅提交未回调，都不算完整完成

### 批量分析标准流程

1. 确认正式接口 `POST /analysis/batch`
2. 规范化 `title`、`symbols`、`parameters`
3. 提交任务时附带 `openclaw_notify`
4. 记录任务关联信息
5. 当前会话不等待
6. 回调到达后再输出排序、筛选和建议

### 持仓分析 / 交易复盘标准流程

1. 先区分“查询现状”还是“发起分析/复盘”
2. 查询类需求走正式查询接口
3. 分析/复盘类需求走正式分析接口，不要用查询接口假装完成分析
4. 如果正式分析接口返回任务标识或需要后续取结果：
   - 走任务闭环
   - 不等待
   - 用正式回调或单次兜底查询获取终态
5. 如果正式接口直接返回最终分析结构，则按官方结果解释

### 交易计划标准流程

1. 先判断当前是生成、评估还是优化
2. 优先调用系统已有生成/评估/优化接口
3. 不要用外部自由发挥替代正式计划接口
4. 若该部署对计划任务返回 `task_id`，同样走任务闭环
5. 若接口同步返回正式结构化结果，则该步骤可以直接视为完成

## 回退策略

如果当前环境缺少 webhook 接收能力，助手应：

1. 明确说明“当前缺少 webhook 或等价回调能力，无法完成标准闭环”
2. 报告缺失项，例如：
   - `OPENCLAW_HOOK_URL`
   - `OPENCLAW_HOOK_TOKEN`
   - `openclaw_notify` 目标信息
   - 可关联会话的 `session_key`
3. 请求用户提供可用回调方式，或明确允许使用文档约定的官方兜底机制
4. 不得自动改走未确认的同步分析接口

兜底只允许：

- 官方文档明确存在的单次状态查询
- 官方文档明确存在的单次结果获取接口

不允许：

- 长轮询
- 猜路径
- 批量穷举接口

## 输出约定

始终用面向用户的语言表达结果，不要直接暴露传输层噪音。

对外汇报时，应明确使用以下状态词，而不要混用：

- 服务可达
- 脚本认证成功
- 任务创建成功
- webhook 回调成功
- 分析完成

成功时，输出应包括：

- 执行了什么动作
- 当前处于哪个状态
- 如果已完成，主要结果是什么
- 关键逻辑或风险点
- 如有需要，给出下一步建议

任务刚提交时，输出应包括：

- 调用了哪个正式任务接口
- 已记录哪些任务关联信息
- 当前为什么不等待
- 后续结果将通过什么方式回到会话

失败时，输出应包括：

- 失败阶段：环境、脚本认证、参数校验、任务创建、回调还是结果获取
- 简明原因
- 最有用的修复建议
